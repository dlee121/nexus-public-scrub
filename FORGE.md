# Forge — autonomous multi-repo development pipeline

Forge is the development pipeline that lives inside Nexus. Given a single
operator instruction ("ship a bandit-tuning experiment to the realtime
platform", "fix the auth-bouncer 403 path on [target-repo-web]", etc.), Forge
plans the work into ordered Linear tickets and drives each ticket through
implement → verify → PR → CI → review → merge → deploy on its own. The
operator gates the plan up front and gates prod deploys; everything in
between is autonomous.

The pipeline is built on **Temporal Cloud** for durable execution. Long
phases (a 90-minute Claude Code IMPLEMENT session, a 35-minute CI wait, a
60-minute deploy) survive worker restarts because Temporal owns the
state machine and replays it.

## Two pieces

| Component         | Path                          | Role                                          |
|-------------------|-------------------------------|-----------------------------------------------|
| `pipeline-worker` | `pipeline-worker/`            | Temporal worker that runs the workflows + activities. Talks to Claude Code, GitHub, Linear, Slack, AWS Elastic Beanstalk. |
| `forge-console`   | `forge-console/`              | Hono-based operator UI / SSE dashboard. Hosts plan-approval and prod-deploy gates, streams live activity events, ingests session transcripts. |

Both processes run on the same EC2 host as systemd units
(`forge-worker.service`, `forge-console.service`) and pull secrets from
AWS Secrets Manager via `forge-secrets.service`.

## State machine

Each ticket is one `pipelineWorkflow` execution. The workflow advances
through these states (defined in `pipeline-worker/src/types.ts`):

```
QUEUED
  → PLAN_REVIEW           # plan critique generated by planReviewActivity
  → IMPLEMENT             # CC session writes the change in a worktree
  → VERIFY                # lint + tests + GPT diff review
  → PR_OPEN               # createPRActivity opens the PR on GitHub
  → CI_WAIT               # block until required checks pass
  ┌→ REVIEW               # waitForExternalReview + Forge /review skill
  │ → ADDRESS_REVIEW      # CC session addresses combined feedback
  │ → CI_WAIT             # re-CI after pushed fixups
  └─ (loop, capped at 5 cycles)
  → MERGE_QUEUE           # mergeActivity merges the PR (squash)
  → DEV_DEPLOY            # devDeployActivity runs `make eb-deploy-dev-tagged`
                          #   (or no-op for autoDeploy repos, e.g. Render)
  → MONITOR               # post-deploy health check + repo-defined script
  → PROD_DEPLOY_GATE      # operator approval via Slack / dashboard
  → PROD_DEPLOY           # runDeployActivity with heartbeats every 30s
  → DONE
```

Failure exits go through `BLOCKED` / `FAILED`. Operator-cancelled runs
exit through `CANCELLED` (a clean status, not an error). `WAIT_FOR_COVERING_DEPLOY`
is used when the deploy coordinator decides that a later merge already
shipped this ticket's SHA — the deploy gets coalesced rather than run
twice.

The `BUGBOT_WAIT` state is deprecated (kept in the union for historical
queries); it was replaced by the `REVIEW` + `ADDRESS_REVIEW` loop.

## Wave / ticket / gate model

Forge accepts an instruction at the *plan* level and explodes it into a
`WavePlan`:

```ts
interface WavePlan {
  instruction: string;                              // operator's prompt
  waves: Array<{ wave: number; tickets: Ticket[] }>;
}
```

A *wave* is a set of tickets that can run in parallel. Waves run
sequentially — wave N+1 starts only after every ticket in wave N reaches
DONE. This is how cross-repo dependencies are expressed: if changes to
`[target-repo-api]` must merge before changes to `[target-repo-web]`, they
sit in earlier waves.

`MultiTicketWorkflow` is the parent that owns this orchestration:

1. Generates the wave plan (or accepts a pre-built one).
2. Sends the operator a Slack ping with the plan summary.
3. Blocks on `planApprovedSignal` (or operator cancel = clean exit).
4. Calls `createLinearTickets` to materialize Linear issues for every
   ticket.
5. For each wave, fan-outs `executeChild(pipelineWorkflow, ...)` for every
   ticket and awaits all of them via `Promise.all`.
6. On any child failure, surfaces the deepest cause through the Temporal
   wrapper chain and Slacks the rollup before re-raising.

Each `pipelineWorkflow` child carries the parent's `WorkflowOptions` (e.g.
`skipProdDeploy=true` for dogfood runs that should stop at MONITOR).

## How Forge uses Claude Code

Three workflow phases each spawn their own dedicated Claude Code session:

- **`implementActivity`** (`pipeline-worker/src/activities/implement.ts`)
  creates a fresh git worktree off the repo's default branch, then runs a
  CC session with the ticket title / description / acceptance criteria
  plus the plan critique from `planReviewActivity`. The session writes
  code, commits, and exits. The activity captures the session JSONL so
  the dashboard can replay the full transcript later.

- **`reviewActivity`** runs the Forge `/review` skill against the open PR
  — a self-review pass that reads the diff and posts a `### Code review`
  comment.

- **`addressReviewActivity`** consumes the combined feedback packet
  (VERIFY-time GPT diff review + Forge self-review + external comments
  from Bugbot / human reviewers) and runs another CC session that pushes
  fixup commits to the PR branch.

All three sessions stream their JSON output line-by-line to the console
via `runCCSessionStreamed` + `emitEvent`, so the operator sees turn-level
activity in real time.

The PR-blocked retry loop (REVIEW → ADDRESS_REVIEW → CI) is hard-capped
at 5 cycles (10 Opus sessions per ticket). Beyond that the workflow
fails non-retryably and a human steps in.

## Other workflows

- **`DeployCoordinatorWorkflow`** — singleton workflow per repo+env that
  serializes deploys. When multiple PRs merge in quick succession, it
  coalesces deploys: if PRs A, B, C all merged before any deploy ran, the
  coordinator deploys C and signals A/B as `coveredByCoalesceSignal` so
  their `pipelineWorkflow` children skip their own DEV_DEPLOY rather than
  redundantly redeploying the same `origin/main` SHA.

- **`PatrolWorkflow`** — long-running quality patrol. Every 6h it runs
  `runPatrolActivity` (a CC session that audits the repo for drift and
  files Linear issues). Uses `continueAsNew` between cycles so event
  history stays bounded.

## Multi-repo registry

The set of repos Forge knows about lives in the top-level `nexus.json`
under `forge.repos.<name>`. Each entry is a `PipelineConfig`
(see `pipeline-worker/src/types.ts`) declaring:

- repo coordinates (`repoOwner`, `repoName`, `defaultBranch`)
- the Temporal task queue (`forge-pipeline`)
- per-repo lint / test / smoke / fix / typecheck commands, with optional
  *diff-aware* variants (`lintDiffCommand` + `lintDiffExtensions`,
  `testDiffCommand` + `testDiffExtensions`) so VERIFY runs only against
  files the branch changed — keeping Forge insulated from the
  pre-existing failure baseline of large legacy repos
- dev / prod deploy commands (`make eb-deploy-...`, `make deploy`, or a
  no-op for `autoDeploy: true` repos like Render-hosted [target-repo-web])
- pointers to per-repo briefs (`templates/forge/*-CLAUDE.md`,
  `templates/forge/*-AGENTS.md`) which the IMPLEMENT session loads as
  context
- post-deploy `monitorScript` and `devHealthCheckUrl` for MONITOR
- optional `seedFixturesScript` for ClickHouse-backed dev environments
  (with a hard guard against ever pointing at a non-dev host)
- per-repo `codeartifact` block when private package mirrors are needed

A ticket carries an optional `repoName` field naming which entry it
targets. Multi-repo waves work because each child workflow resolves its
own `getRepoConfig(ticket.repoName ?? DEFAULT_REPO_NAME)`.

## Operator surfaces

- **Plan-review gate** — Slack ping + dashboard button. Approve =
  `planApprovedSignal`. Reject = workflow cancellation (clean
  `CANCELLED` status).
- **Prod-deploy gate** — same shape, signals `prodApprovedSignal` /
  `prodRejectedSignal`. Reject is a valid outcome (dev-only ship, end
  workflow with `DONE`).
- **Cancel anytime** — operator clicks reject anywhere; the workflow's
  `condition()` raises `CancelledFailure`, the workflow records
  `CANCELLED`.
- **Slack pings** at every phase boundary (best-effort, never propagate
  failures — Slack outage doesn't fail a deploy).

## Reading order

1. `pipeline-worker/src/types.ts` — `Ticket`, `WavePlan`, `PipelineState`,
   `WorkflowOptions`, `PipelineConfig`. Defines the contracts every
   workflow / activity speaks.
2. `pipeline-worker/src/workflows/MultiTicketWorkflow.ts` — the parent
   orchestrator, plan-approval gate, wave fan-out.
3. `pipeline-worker/src/workflows/PipelineWorkflow.ts` — the per-ticket
   state machine. Single longest file in the repo and worth reading
   end-to-end; every other file is in service of it.
4. `pipeline-worker/src/workflows/DeployCoordinatorWorkflow.ts` — the
   deploy coalescing model.
5. `pipeline-worker/src/activities/implement.ts` — how a workflow phase
   becomes a Claude Code session.
6. `forge-console/src/server.ts` and `forge-console/src/routes/` — the
   operator UI: workflow list, transcript SSE, gate buttons, event
   ingestion.
7. **`data/task5-transcript.md`** — fully-redacted log of a real
   three-repo, multi-ticket Forge run end-to-end. The clearest single
   artifact for understanding what an actual execution looks like.
